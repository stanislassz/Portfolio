GLPI

Ce projet m’a permis de mettre en pratique la compétence B1.2 en assurant la gestion des incidents et des demandes d’assistance via GLPI, en configurant les utilisateurs et les profils, en organisant le suivi des tickets et en contribuant à l’amélioration du support informatique et du traitement des besoins des utilisateurs.


Voici un exemple concret de gestion de ticket avec GLPI :

  • Un utilisateur signale via le portail un problème d'accès à sa messagerie (création du ticket).
  • Le technicien attribue le ticket au groupe "Support Poste de travail", ajoute une priorité "Moyenne" et renseigne la catégorie "Messagerie".
  • Le technicien effectue des vérifications (reset de mot de passe, test de connexion) et documente chaque action dans le fil de suivi du ticket.
  • Après résolution (par exemple réinitialisation du compte et test de réception/envoi), le technicien passe le ticket en état "Résolu" et précise les actions réalisées.
  • Le ticket est ensuite clôturé après confirmation de l'utilisateur ou automatiquement après validation, et les informations sont conservées pour l'historique et l'amélioration continue du support.
Cette démarche illustre la traçabilité, la communication avec l'utilisateur et l'organisation du traitement des incidents via GLPI. D'autres outils : MantisBT, OTRS, Jira Service Management, Freshdesk, Zendesk, ServiceNow, etc.

Cette capture montre le dashboard de l'application C#, utilisé pour centraliser les données du projet et suivre l'organisation de l'interface.


Le projet C# m'a permis de structurer une application avec une interface utilisateur claire, en utilisant les bonnes pratiques de développement pour assurer la maintenabilité et l'évolutivité du code. J'ai ainsi pu gérer efficacement le patrimoine informatique du projet tout en développant mes compétences techniques.

Voici les éléments clés de l'organisation du projet C# :

  • Structure MVVM : séparation claire entre Model (données, accès aux services), ViewModel (logique, commandes, observables) et View (interface, liaison de données).
  • Model : classes métier + gestion de la persistance (repository, DTOs, accès base de données ou API).
  • ViewModel : expose les propriétés observables, implémente les commandes, gère la validation et la communication asynchrone avec le Model.
  • View : XAML/HTML/CSS ou équivalent lié au ViewModel via bindings; contient uniquement le code d'affichage et peu ou pas de logique métier.
  • Tests : unitaires pour le ViewModel et le Model; tests d'intégration pour les services.
  • Utilisation de GitHub : dépôt dédié avec branches (main, develop, feature/*), pull requests, issues et templates pour PR/issue.
  • Documentation : README.md expliquant l'architecture, instructions d'installation, exemples d'utilisation, conventions de code; docs supplémentaires dans /docs (diagrammes, flow, API).

Ce projet m'a permis de consolider mes compétences en développement d'applications, en gestion de projet et en organisation du code pour assurer la qualité et la maintenabilité du patrimoine informatique. J'ai ainsi pu structurer efficacement le projet tout en développant mes compétences techniques et professionnelles.
